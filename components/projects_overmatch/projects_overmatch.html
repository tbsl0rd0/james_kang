블리자드사의 오버워치가 출시된 이후로 한국에서 어느정도 인기를 얻었습니다.  오버워치는 팀 단위로 하는 게임이기 때문에 불특정 다수를 대상으로 오랫동안 호흡을 맞춰 플레이할 팀을 모집하는 시스템이 필요하다고 생각했습니다.  오버워치 인벤에 이러한 시스템이 존재했지만 게시판 형태로 존재했기 때문에 불편해 보였습니다.
Since blizzard's overwatch was released, it has gained some popularity in korea.  Overwatch is a team-based game, so it needs a system that recruited a team for cooperating each other for a long time.  The system exgists in overwatch inven, but it looks uncomportable because it exists in the form of a bulletin board.

overmatch (overmatch.co.kr)
overmatch github

Overmatch has the following characteristics.
- 회원가입 필요없이 배틀넷 계정으로 로그인합니다. (OAuth 2.0 프로토콜)
- 배틀태그를 클릭하면 여러가지 편리한 기능을 이용할 수 있는 컨텍스트 메뉴가 뜹니다.
- 배틀태그와 함께 전적을 항상 표시함으로서 해당 유저의 성향을 쉽게 파악할 수 있습니다.
- 공개채팅, 구직란, 파티매칭 등에 웹소켓을 사용하여 실시간 데이터 통신이 가능하게 합니다.
- Users login with battlenet account without signing up.
- Clicking a battletag brings up a context menu where users can use variable useful functions.
- Users can easily identify the other's stat by always showing the stat aside the battletag.
- It enables real-time data communication by using websocket for public chat, job search, and party matching.
프로젝트 목적과는 별도로 이 프로젝트는 혼자서 웹앱을 기획, 제작, 홍보, 운영해보고 실력을 다지는 목적도 어느정도 있습니다.  특히, 짜임새 높은 MEAN 스택 SPA 앱을 제작하는데 의의를 두었습니다.
Apart from the purpose of the project, the project's private purpose is planning, producing, promotion and operating web apps on my own and to improve my skills.  Especially, I put an important point on making high-structural MEAN stack SPA app.

배틀넷 계정을 이용한 로그인
Login using the battlenet account
OAuth 2.0 표준 프로토콜 사용하는 passport.js를 이용해서 배틀넷 계정으로 로그인하는 시스템을 구현했습니다.  아직 배틀넷 API가 오버워치 쪽은 구현이 되어있지 않아 배틀태그만 얻어올 수 있습니다.  현재 전적이나 레벨, 랭크 티어 등은 오버워치 공식 사이트를 파싱해서 가져오고 있습니다.  배틀넷 API에서 오버워치를 정식으로 지원하면 파싱할 필요 없이 API로 가져올 수 있기를 기대합니다.
I have implemented a system that users login using battlenet account using passport.js, which uses the OAuth 2.0 standard protocol.  The battlenet API is not yet implemented on overwatch, so the overmatch can only get the battletag.  Currently, level, rank tiers, etc are parsed from the overwatch official site.  If the battlenet API supports overwatch, I expect to be able to fetch them with the API without having to parse them.
공개채팅
Public chat
공개채팅은 자유로운 주제로 대화를 나누는 곳입니다.

공개채팅은 기본적으로 웹소캣을 이용하여 구현했습니다.  공개채팅 탭에 있는 동안 채널에 가입이 되고 다른 탭에 있는 동안 채널에서 떠나므로 공개채팅 탭에 있는 유저들에게만 채팅 정보를 브로드캐스팅합니다.
Public chat is where users talks with a free conversation topic.

Public chat is basically implemented using web sockets.  Because a user is subscribing when is in the public chat tab, a user is unsubscribing when is in the other tab, the chat information only broadcasts to the user in public chat tab.

Job search board
파티에 들어가고자 하는 유저들이 구직란을 통해 자신을 어필할 수 있는 곳입니다.  코멘트와 포지션, 마이크 사용 여부 등을 입력하고 포스팅을 하면 다른 유저들이 이 포스팅을 볼 수 있고 오버워치 게임 내에서 직접 친구 추가를 하거나 파티매칭에서 방을 만들고 초대를 할 수 있습니다.

구직란도 웹소켓을 통하여 데이터가 실시간으로 동기화되도록 했습니다.
The job search board is a place where users who want to enter a party can appeal themselves  Entering Comments, positions and whether you use a microphone, and posting, other users can see this post and then add friends directly within the overwatch, or create a room in the party matching room board and invite them.

Job search board also let the data be synchronized in real time through the websocket.

동영상 게시판
Videos board
동영상 URL을 등록하고 기호에 맞는 동영상을 여러 유저들과 공유하며 댓글로 소통할 수 있는 곳입니다.

다른 동영상 게시판과 차별을 두기 위해서 다음과 같은 기능을 두었습니다.
It is a place where you can register video URLs, share the videos that match preferences with various users, and communicate with comments.

In order to differentiate it from other videos boards, it has the following special functions.
- 동영상 포스트에 타이틀과 가깝게 베스트 댓글을 표시합니다
- 동영상의 +1/-1이나 댓글의 +1/-1은 실시간으로 동기화됩니다.
- 댓글 자체도 마치 채팅을 하는 것처럼 실시간으로 동기화됩니다.
- Show best comments close to titles in video posts
- Video + 1 / -1 or comment + 1 / -1 is synced in real time.
- The comments themselves are synced in real time as if they were chatting.
현재 동영상 게시판 활성화를 위해서 자동으로 10분에 한번씩 다른 커뮤니티 사이트에서 게시물을 파싱하여 등록하도록 하였습니다.  코드 관리와 운영 비용 절감을 위해서 AWS lambda를 이용하여 스케줄링했습니다.
Currently, to activate the videos board, it automatically registers the posting every 10 minutes by parsing other community site.  I used AWS lambda to schedule this to manage its code and cut down the operating costs.

웹팩을 이용한 컴포넌트 번들화
Component bundling with webpack
웹팩의 로더를 통해서 위의 컴포넌트 구조의 js 파일에서 html 파일을 스트링으로 읽어서 템플릿으로 지정하고 less 파일을 컴파일하여 css로 지정합니다.  이렇게 1차적으로 번들화된 컴포넌트 파일을 묶어 최종적인 번들 파일을 만들어 냅니다.

--watch 옵션을 통해 이런 일련의 작업들은 코드가 변경될 때마다 자동으로 실행됩니다.
Through the webpack loader, it reads the html file as a string and designate it as a template, and compile the less file and load it into css from the js file of the component structure above.  These primary bundled component files are bundled together to produce the final bundle file.

With the --watch option, this set of tasks is automatically executed whenever the code changes.
걸프를 이용한 번들 파일과 라이브러리의 통합
Integration of bundle files and libraries using gulf
걸프는 웹팩을 통해서 만들어진 번들 파일과 필요한 라이브러리들을 한데 묶어 minify를 하고 하나의 파일로 통합합니다.  최종적으로 자바스크립트와 관련된 파일은 js.min.js로, CSS와 관련된 파일은 css.min.css로 만들어집니다.  이렇게 하나의 파일로 통합했을 때는 요청을 여러번 하는 대신 하나의 파일만 요청하게 하므로서 과부하를 막을 수 있고 관리의 편의성이 있습니다.

걸프도 마찬가지로 코드가 변경될 때마다 watch 태스크를 통해서 자동으로 실행됩니다.
Gulf bundles bundle file made with webpack and necessary libraries, minifies and integrates it into one file.  Finally, the javascript-related files are created as js.min.js, and the CSS-related files are created as css.min.css.  When combined into a single file, it can prevent overloading by making only one file to be requested instead of multiple requests.

The gulf also runs automatically through the watch task whenever the code changes.

Images or js.min.js and css.min.css are not served directly from EC2.  Because S3 is less overloaded and has a faster speed, resources other than server code are more advantageous in S3.
user_data의 활용
Using user_data
EC2의 인스턴스가 생성될 때 제일 처음 실행되는 쉘 코드가 user_data입니다.  EC2에 직접 접속하여 실행 환경을 만들고 코드를 배포하는 방법 대신에 user_data를 통해서 실행 환경을 설정하고 코드를 배포하는 방법을 선택했습니다.  AMI를 따로 만들어 둘 필요가 없고 웹이 업데이트 되었을 때 새로운 버전의 코드를 배포하기가 편리하기 때문입니다.

다음은 오버매치의 user_data입니다.
The first shell code that runs when an instance of EC2 is created is user_data.  Instead of connecting directly to EC2 to create a runtime environment and deploying the code, I chose user_data to do them.  This is because it is not necessary to create AMI and it is convenient to distribute the new version of the code when the web is updated.

Here is an overmatch user_data.
Bash 
#!/bin/bash

cd /home/ec2-user

wget https://nodejs.org/dist/v6.9.4/node-v6.9.4-linux-x64.tar.xz
tar xf node-v6.9.4-linux-x64.tar.xz
rm node-v6.9.4-linux-x64.tar.xz
mv node-v6.9.4-linux-x64 node

export PATH=$PATH:/home/ec2-user/node/bin
export NODE_ENV=production

yum update -y

yum install -y git
yum install -y awslogs

npm install -g forever

iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-ports 3000
service iptables save

git clone https://github.com/tbsl0rd0/overmatch.git
cd overmatch
npm install

forever start bin/www

sed -i 4s/.*/'region = ap-northeast-2'/ /etc/awslogs/awscli.conf

sed -i 121,127d /etc/awslogs/awslogs.conf

echo [forever_logs] >> /etc/awslogs/awslogs.conf
echo log_group_name = overmatch >> /etc/awslogs/awslogs.conf
echo log_stream_name = forever_logs >> /etc/awslogs/awslogs.conf
echo file = /root/.forever/*.log >> /etc/awslogs/awslogs.conf

service awslogs start
먼저 실행 환경에 필요한 프로그램을 다운로드하고 설치하며 환경변수를 설정합니다.  아이피테이블을 이용하여 TCP 80번 포트를 TPC 3000번 포트로 리다이렉트합니다.  그리고 깃허브에서 서버 코드를 클론하고 노드 모듈을 설치합니다.  그 다음 포에버로 서버를 실행하고 AWS cloudwatch logs에 포에버 로그를 전송하기 위한 파일 설정을 합니다.  끝으로 awslogs 서비스를 실행하고 서버를 구동시킬 모든 준비를 마칩니다.
First, download and install the necessary programs for the runtime environment, and set environment variables.  TCP port 80 is redirected to TPC 3000 port using iptables.  Then clone the server code from the github and install the node modules.  Then run the server with forever and set up the file to send the forever logs to AWS cloudwatch logs.  Finally, ready to run the awslogs service and start running the server.

- Server-side rendering, not client-side rendering (speeding up)
- One-way binding, not bidirectional binding (speeding up)
- Typescript formally supported to use ES6, 7 (development / collaboration convenience)
레스트 API
RESTful API

As a result, I failed to apply the RESTful API to the overmatch.  It can easily be applied to an environment that simply CRUD the resources, but the overmatch has been harder to apply.  There are a lot of cases where a resource is updated, and I would like to give a specific name to each one of these updates, but the RESTful API was inconvenient in this regard.  I will learn a lot more about the RESTful API and how to deal with resources before I apply it.

다음과 같은 코드는 IE에서 문제를 일으킵니다.

<input type="text" ng-model="message">

printing 'gx5u3s8m#3373'

몽고DB의 _id의 문제
The problem of the _id of the mongodb
몽고DB의 _id는 몽고 DB에서 자동으로 생성하는 식별 아이디로 오브젝트로 저장됩니다.  이 _id가 에이잭스를 통해서 클라이언트로 전달될 때는 문제가 생기지 않지만 웹소켓을 통해서 전달될 때는 문제가 생깁니다.  개발환경인 윈도우에서는 별다른 문제가 생기지 않았지만 운영환경인 리눅스에서는 이 _id가 웹소켓으로 전달될 때 문자열로 변형되지 않고 오브젝트 그대로 전달되는 현상이 발생했습니다.  이 문제는 uuid 라이브러리를 통해서 _id 대신에 따로 부여하 uuid를 사용함으로서 해결했습니다.
The _id of mongodb is stored as an object with the ID that is automatically generated from mongodb.  There is no problem when this _id is passed to the client via ajax, but it will be a problem when passed through the websocket.  On windows, which is a development environment, there was no problem, but in linux, which is an production environment, when this _id was passed to a websocket, it was transferred as an object instead of being transformed into a string.  This problem was solved by using the uuid library to give uuid instead of _id.

IE에서 $http 모듈의 캐쉬 문제
Cash problem of the $http module in IE
처음 오버매치 웹사이트 접속하게 되면 세션을 얻어오는 에이잭스를 수행합니다.  이 에이잭스는 앵귤러의 $http 모듈을 주입해서 수행합니다.  그러나 이 $http 모듈의 기본 설정으로 캐쉬를 하게 되어 있습니다.  문제는 크롬에서는 상태 코드 200으로 세션을 잘 얻어오는 모습을 보였지만 IE에서는 상태 코드 302로 캐쉬된 세션을 사용하는 현상이 나타났습니다.  그렇기 때문에 IE에서는 처음 접속할 때 세션을 얻어오지 못하고 새로고침을 했을 때 얻어오는 현상이 일어났습니다.  그래서 다음과 같이 $http 모듈의 기본 설정에서 캐쉬를 사용하지 않도록 했습니다.
The first time an user connects to the overmatch website, Ajax gets the session.  This Ajax is accomplished by injecting the angular $http module.  However, the default setting for this $http module is to cache.  The problem was that Chrome was getting a good session with a status code of 200, but in IE it was using a cached session with status code 302.  Therefore, when an user first connected in IE, it could not get a session and get it when the user refreshes it.  So I decided not to use the cache in the default setting of $http module as follows.
JS 
angular.module('init', [])
.component('init', {
  controller: function() {}
})
.config(function($httpProvider) {
  $httpProvider.defaults.headers.common = {
    pragma: 'no-cache'
  };
});
